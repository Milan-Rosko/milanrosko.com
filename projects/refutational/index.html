Summary

REFUTATIONAL LEARNING

Refutational Learning is a learning schema in which training proceeds by eliminating explicit refutation witnesses against a semantic object. A learner is not primarily moved through a loss landscape; it is repaired in response to exhibited failures. The process terminates, relative to a declared finite obligation set, when no remaining witness refutes the object.

On Refutational Learning

The central contribution is a constructive account of learning as semantic repair. The process begins when the current interpretation fails a declared margin condition. That failure is exposed by a refutation witness: a finite example/class pair that identifies the specific margin violation.

The learner responds with a local integer repair. Repeating this step produces a finite trace in which each state follows from an explicit witness and repair. When no declared refutation witness remains, the interpretation has reached closure on the finite table.

Glossary

Semantics lock: the fixed preprocessing, feature schema, scoring rule, readout rule, and scan schedule.

Witness table: the fixed finite collection of examples over which the declared obligations are checked.

Refutation witness: a concrete finite example/class pair showing that the current state violates the declared margin condition.

Repair: the integer update triggered by such a witness.

Closure: the condition that no declared finite refutation witness remains.

Trace: the replayable sequence of states produced by repairs.

The comparison with Gradient Descent therefore concerns two conceptions of training: explicit repair of witnessed semantic failure, and numerical descent over a scalar objective.

This is not a benchmark claim. The comparison isolates what each training trace makes visible.

SGD is included because it compresses failure into a numerical loss signal. Refutational Learning keeps the failing obligation explicit as a refutation witness.

Note. The demo is easiest to read after the short technical orientation below.

Both are implemented in JavaScript and judged by the same locked prediction rule $\arg\max\nolimits^{*}$, but the aim is analytic rather than competitive. The shared interface isolates the difference between repairing a finite interpretation in response to a refutation witness and updating coordinates according to a differentiable objective.

The extensional interface remains fixed and each trace runs under its declared scan schedule. The important difference is not only how the update is computed, but what kind of evidence the update leaves behind.

(i) We fix the witness table, the feature schema (bias, unary facts), the number of classes, and the scan order used by each method. Each cycle therefore has an unambiguous definition and can be replayed exactly.

(ii) Refutational Learning locates an explicit refutation witness, performs a local repair on the sparse integer table $W$, and propagates its semantic consequences through postings into a cached score matrix $S$. Gradient Descent instead performs SGD updates on softmax cross-entropy with floating weights, while prediction is still read out by the same locked $\arg\max\nolimits^{*}$.

What the trace shows. The Refutational Learning trace exposes each refutation witness, repair, and resulting state. The GD trace reports loss and accuracy computed under the same locked readout.

We start with linearly separable datasets, e.g. $\mathrm{NAND/OR/AND}$ where finite-table closure is typically reachable. Alternatively, we can examine “harder” cases, some of which are not separable in this setting, e.g. $\textrm{XOR}$ or $\textrm{Parity-of-3}$.

These cases are useful because failure is informative: the trace can show persistent refutation witnesses rather than falsely certifying success.

The Boolean presets show the repair logic in miniature. The EMNIST notebook applies the same discipline to a larger finite witness table.

The repair loop

  1. Evaluate the current semantic state.
  2. Find the earliest refutation witness to a margin violation.
  3. Repair the state locally.
  4. Check whether any finite refutation witness remains.

For state $\Sigma_t=(W_t,S_t)$, the next witness is selected by its earliest example index $i^*$; $c^*$ is the strongest competing class under the fixed tie-broken readout:

$$ \begin{array}{l} \Sigma_0 := (W_0,S_0) = (0,0),\\[0.6ex] \Sigma_{t+1} := \mathsf{Repair}_{\gamma,\Delta}(\Sigma_t),\\[0.8ex] i^* = \min\{\, i < N \mid \exists c\neq y_i:\ S(i,y_i) < S(i,c)+\gamma \,\},\\[0.4ex] c^* = \arg\max\nolimits^{\!*}_{c\neq y_{i^*}}\, S(i^*,c). \end{array} $$

The trace is not merely a visualization. It is the finite history from which the terminal state and its checks can be recomputed.

Formal setup

Fix deterministic preprocessing, a feature map $\Phi$, a finite witness table, and a stable tie-broken $\arg\max^{*}$ under a semantics lock $\mathcal{L}$. Let the learner state be a sparse integer weight table $W$ together with a cached score matrix $S$. Training selects the earliest refutation witness to a margin violation, applies a local repair, and terminates when every declared margin obligation is satisfied on the table.

In the current implementation, the evaluator can be viewed as a lambda-definable perceptron under the declared semantics.

Aims:

  • Object. Learning transforms a finite interpretation with explicitly declared semantics.
  • Method. Each update eliminates a concrete refutation witness while preserving decidable invariants.
  • Gate. Accept the repaired evaluator only when finite validation under the lock $\mathcal{L}$ finds no remaining refutation witness.

The lock fixes preprocessing, feature extraction, tie-breaking, scan order, and evaluation criteria. These are the usual hidden degrees of freedom removed by the construction.

Finite means that the obligation set is bounded and explicit. Replayable means that the trace can be re-executed. Checkable means that its state predicates can be recomputed without trusting the trainer.

Training manipulates an explicit, finitary state

$$ \Sigma \;\equiv\; \{\;W, S, ...\;\}, $$

where $W$ is a sparse integer weight table and $S$ is a cached score matrix (maintained via postings). This is the current prototype’s state representation, not a limit of the general idea. After the repair loop is understood, each snapshot can also be checked by decidable predicates with diagnostic witnesses on failure:

$$ \begin{array}{l|l} \mathsf{WF}(\Sigma) & \text{Well-formedness of the finite representation}\\[0.4ex] \mathsf{CACHE.ok}(\Sigma) & \text{Cache correctness}\\[0.4ex] \mathsf{FULL.ok}(\Sigma) & \text{Zero-violation / margin closure by } \arg\max^{*}\\ \end{array} $$

These are not performance metrics. They are state predicates. They say whether the finite representation is well-formed, whether the cached scores still mean what they claim to mean, and whether the declared finite margin obligations have been closed.

WF says the finite state has the right shape: weights, examples, features, classes, and cache entries line up.

CACHE.ok says the fast cached scores agree with the slower definitional recomputation. This separates the learned object from an accidental implementation shortcut.

FULL.ok means: under the locked readout and margin rule, the finite table no longer contains a witnessed failure.

When a predicate fails, the system returns a concrete diagnostic witness: an explicit index and class showing a cache discrepancy, or, for FULL.ok, the earliest refutation witness to a margin violation. When FULL.ok holds, the state has reached closure. Together with well-formedness and cache correctness, this supplies a finite certificate about the artifact under $\mathcal{L}$, not a global generalization claim.

Operationally this resembles error-driven methods (perceptron-like in spirit), but its interpretation is different. A gradient is not taken as evidence of direction in a pre-given topological landscape. The learner instead receives a refutation witness to a declared semantic obligation and performs a repair whose effect is explicitly propagated. Learning is therefore a sequence of witness eliminations over a named object. Implementation substitution is considered only after the resulting state has passed its declared checks.

Let $\textsf{E}_{\mathsf{ref}}$ be a reference evaluator induced by the lock $\mathcal{L}$ (e.g. recomputation from $\Phi$ and a declared scoring rule), and let $\textsf{E}_{\mathsf{learn}}$ be the learned evaluator encoded by a terminal snapshot $\Sigma_T$. Here, “replace” means implementation substitution: using $\textsf{E}_{\mathsf{learn}}$ where $\textsf{E}_{\mathsf{ref}}$ would otherwise be evaluated. Before that substitution, a bounded comparison must establish semantic agreement on the locked finite table, such as

$$ \forall i < \mathrm{N} : \textsf{E}_{\mathsf{learn}}(x_i) = \textsf{E}_{\mathsf{ref}}(x_i) $$

This obligation establishes equivalence only on the declared finite table; it does not identify the two implementations or justify behavior outside the lock. Acceptance of the learned state is a separate question governed by the declared closure and state predicates. When the substitution check fails, it returns the concrete failing index $i$.

Limits

The prototype is intentionally simple extensionally. Its role is to isolate the training discipline: deterministic refutation, finite repair, replayable trace, and explicit closure.

The project does not dissolve the black box. Once learning amounts to selecting a program in an expressive language, intensional structure remains underdetermined by behavior. If the underlying problem contains underdetermination, intractable minimization, or undecidable properties, additional model complexity does not by itself remove that core.

Research direction

Refutational Learning may be well suited to regimes where a semantic object is repeatedly repaired in response to new refutation witnesses. The broader question is which parts of this training discipline survive beyond the prototype’s finite-table representation.
